Split View: Past + Future

Latest Feedback

  • There's persistent interest in grouping multiple split panes together into a single unit that can be brought up or dismissed with a single action
    • This might look like a "double tab" that includes two tabs
    • This might be accomplished by allowing you to "weld" split panes together
  • The floating (x) icon is a common source of frustration
    • Folks confuse the (x) for an in-webpage UI element ‚Äî maybe it should be moved outside the web contents?
    • The (x) occasionally blocks UI elements ‚Äî maybe it should be draggable?
    • It's unclear if the (x) closes the tab or closes the pane
      • The (x) tooltip ‚Äî "Close" ‚Äî should be clearer. The icon could also be changed.
  • Members commonly ask for horizontal split view, not just vertical split view
  • Occasionally, folks ask that a pane's width automatically grow when it becomes the focused pane
  • There's confusion around the "lock tab" entry point.
    • Based on team feedback, we allowed you to lock a split pane even when it's the only pane (i.e. you aren't in "split view"). This "lock tab" menu item and Type Travel action is a source of confusion
  • The "drag from the right edge of the browser to create a split pane" gesture conflicts with horizontally resizing the window. This has been a source of complaints. My hunch is that we can just remove this entry point, but I'd like to wait until we have metrics.
  • Very positive feedback from Loren and others on "locked panes"

April/May 2021 Revisions

Goals

  • Make it clearer which Split Pane is being acted upon
  • Make split panes feel more permanent and predictable by allowing users to "lock" a particular split pane
  • Make the animation when creating a split pane clearer and more satisfying
  • Understand the most common Split View entry points so we can further refine them, and possibly remove less-used entry points.

"Avocado" Cycle

  • Rewrite split view for smoother interactions
  • Iterate on the (x) buttons
  • Make the (x) appear on whichever pane is hovered
  • Make split view state persist per-LFM
  • Add keyboard shortcuts (control+shift+1,2,3 control+shift+(plus/minus) for switching between split panes and adding/removing them
  • When resizing split view panes, snap the cursor to the 50% point when close to the middle

"Bell" Cycle

Roughly in priority order

  • Closing a tab in a split pane should not close the split pane ‚Äî it should just show the empty state (Based on this feedback)
  • Add the ability to "lock" a split pane so that you cannot navigate away from a particular tab
  • Fix some bugs around the command bar and new split panes
    • When you use type travel or a keyboard shortcut to create a new split pane, we should focus the command bar
    • When you drag to create a split view, we should show the command bar over the new pane, not the old pane (bug)
  • Address user feedback that it's not clear which tab is being acted upon when switching or opening new tabs
    • Increase the width of the blue border around the selected pane
    • Center the Floating Command Bar (and type travel) above the current split pane ‚Äî not the whole window (Adena's suggestion)
      • We should still have >50px padding around the floating panel
    • Idea: when the user hovers over a tab in the Sidebar while using Split View, gently "pop" the pane that it'll open in
    • Show the CMD+Shift+C toast (and related toasts) in the top-right corner of the current split pane, not in the top-right corner of the window
  • Add analytics to track split view entry points. Are people dragging the bar on the right side of the window to create a split view pane? Are they dragging tabs out from the Sidebar? Are they using the right-click menu? Are they using keyboard shortcut?
  • Fix unclear behavior around the "open in Split View" right-click menu action when there's already a split view
    • If there's already a split view, the "open in Split View" menu item should be replaced with "Open in New Split Pane". For each split pane besides the current, add an item to open the link in that pane (e.g. "Open in Split Pane 2")
  • Improve the "empty state" of a split view pane [Needs Design]
    • When a split pane is empty, the (x) control should always be visible ‚Äî even if the pane isn't hovered
  • Add a blurred overlay when split view panes are resizing, so that we don't jankily animate the width of web contents
  • Improve the animation when dragging a tab into Split View
  • Make it possible to drag to reorder split panes [Needs Design]

November 2020 Revisions

Proposal

This revision aims to make Split View clearer by removing the concept of a ‚Äúpersistent pane.‚Äù

Changes

  • There is no concept of one pane being ‚Äúpersistent‚Äù. Instead, in Split View, there are just 2 (or more) panes per window.
  • At any given time, one of the panes will be ‚Äúfocused.‚Äù Clicking a pane focuses it. The focused pane has a blue border around it.
  • Each pane has its own sidebar UI state, with its own selected tab, scroll position, set of expanded items, and list focus mode state. The sidebar shows the state of the focused pane.
  • If you hit CMD+W or close the current tab in a pane, that pane will select another tab to be current, just like non-split windows do.
  • The overlay on the persistent split view pane is replaced with a ‚Äúcontrols‚Äù overlay that can appear on either pane when split view is active.
    • It contains only an (x) button to close the split pane
    • [Optional] It may also contain a ‚Äúmove‚Äù button that allows you to drag the pane to the other side of the split view
  • [Optional] using the hover controls, you‚Äôll be able to ‚Äúmove‚Äù panes to the other side of the window.
  • a new entry point for Split View ‚Äî dragging from the right edge of the browser (see below)

What stays the same

  • Entry points ‚Äî you can still enter split view by:
    • dragging a tab onto either side of the browser‚Äôs content area
    • right-clicking a tab and choosing ‚ÄúShow in Split View"
    • right-clicking a link and choosing ‚ÄúOpen in Split View"

Switching Focused Panes

  • Clicking within a pane, or clicking the tab associated with it, ‚Äúfocuses‚Äù the pane.
  • You can also use a keyboard shortcut [TBD] to focus the other pane.
